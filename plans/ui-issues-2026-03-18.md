# Plan: UI Issues

_Started: 2026-03-18_
_Closed: 2026-03-18_

## Context

Three UI bugs related to chat and project isolation in the sidebar/chat view:

1. **New chats not clickable in sidebar** — newly created chats appear in the sidebar as a non-clickable spinner `<div>` while pending. Users can't switch back to a new in-progress chat until it finishes (or the app restarts).
2. **Chats leaking across project tabs** — the pending spinner for a new chat in Project A also shows up in Project B's sidebar.
3. **Input field shared across projects** — text typed in the chat input persists when switching to another project tab.

## Root Causes

### Issue 1: New chats not clickable
`SessionBrowser.tsx:121–131` renders the pending new session as a `<div>`, never a `<button>`.
While `pendingNewSession.sessionId` is null (before `claude:init` arrives) there is no ID to navigate to, so non-clickable is correct.
But once `sessionId` IS set, the spinner can persist if the session hasn't appeared in `projectSessions` yet (query hasn't re-fetched or cwd mismatch). During this window the user is stuck: the spinner is non-clickable and the session isn't in the list yet.

### Issue 2: Cross-project spinner leak
The condition that shows the spinner (`SessionBrowser.tsx:121–131`) does not check `pendingNewSession.projectPath === activeProjectPath`. So a pending session from Project A always makes the spinner appear in every project's sidebar.

### Issue 3: Input not scoped per project
`ChatInput` keeps its text in `useState("")` (local to the component). React does not remount `ChatView` (or `ChatInput`) when `activeProjectPath` changes — only the store values change. The local state persists.

## Fixes

### Fix 1 — Make pending spinner clickable once sessionId is known
**File:** `src/renderer/components/sessions/SessionBrowser.tsx`

Add `setActiveSession` from the sessions store. Change the pending indicator from a static `<div>` to a `<button>` that is:
- `disabled` when `pendingNewSession.sessionId` is null (not yet navigable)
- clickable (calls `setActiveSession`) when `sessionId` is known

```tsx
// Add to store selectors:
const setActiveSession = useSessionsStore((s) => s.setActiveSession);

// Change the pending indicator:
{pendingNewSession && pendingNewSession.projectPath === activeProjectPath && (  // <-- Fix 2 also applied here
  !pendingNewSession.sessionId ||
  !projectSessions.some((s) => s.sessionId === pendingNewSession.sessionId)
) && (
  <button
    disabled={!pendingNewSession.sessionId}
    onClick={() => {
      if (pendingNewSession.sessionId) {
        setActiveSession(pendingNewSession.sessionId, pendingNewSession.projectPath);
      }
    }}
    className="flex w-full items-center gap-2 rounded-md border-l-2 border-accent-blue bg-muted px-3 py-2 text-left disabled:cursor-default"
  >
    <Loader2 className="h-3.5 w-3.5 shrink-0 animate-spin text-accent-blue" />
    <p className="line-clamp-1 flex-1 text-sm text-foreground">
      {pendingNewSession.firstPrompt}
    </p>
  </button>
)}
```

### Fix 2 — Scope pending spinner to correct project
**Already included in Fix 1** — the `pendingNewSession.projectPath === activeProjectPath` guard is added to the condition.

### Fix 3 — Reset chat input on project switch
**File:** `src/renderer/components/layouts/AppLayout.tsx`

Add `key={activeProjectPath ?? ""}` to `<ChatView>`. This forces a remount (and full state reset) whenever the active project changes:

```tsx
<ChatView key={activeProjectPath ?? ""} />
```

The `ChatView` mounts/unmounts cleanly: IPC listeners re-register on mount, live messages are in the Zustand store (not component state), and the history query re-fetches on mount.

## Decisions

- **Fix 1+2 in one change** — the project-path guard and the clickable fix both touch the same condition block, so combine into a single edit.
- **Key on `activeProjectPath` only (not `activeSessionId`)** — user only reported project-level input leak. Keying on `activeSessionId` would also reset input on session switch, which may be fine but is more aggressive than reported. Revisit if needed.
- **Don't add optimistic session to `projectSessions`** — simpler to make the pending item clickable than to inject a fake entry into the derived list.
